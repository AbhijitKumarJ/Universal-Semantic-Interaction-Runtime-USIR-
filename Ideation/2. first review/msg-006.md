 This is where the architecture gets genuinely interesting — because voice isn't just TTS bolted onto a screen interface. It requires a fundamentally different projection pipeline. The screen renderer maps entities to *visual space*. The voice renderer maps them to *temporal sequence*.

---

## The core problem voice introduces

A screen can show ten entities simultaneously. A voice interface can present exactly one thing at a time, in order, and the user can't "glance" at something — they have to wait for it. This means the voice renderer has to make decisions the screen renderer never has to make:

- What is the **one thing** most worth saying right now?
- How do you express a *relationship* (file → function → error) in linear audio?
- How do you handle disambiguation without visual labels?
- When do you speak unprompted vs. wait to be asked?

These aren't TTS problems. They're **projection and sequencing** problems. TTS is just the last step.

---

## The voice-specific projection

The `ProjectionEngine` from the screen path runs the same scope and rank logic. But before handing off to the voice renderer, it passes through a `VoiceProjector` that reshapes the `ProjectedView` for linear consumption:

```typescript
// packages/runtime/src/projection/voice-projector.ts

export class VoiceProjector {

  project(view: ProjectedView, memory: InteractionMemory): VoiceScript {
    
    // 1. Determine utterance budget — how much should we say?
    const budget = this.computeBudget(view);
    
    // 2. Build the primary utterance from the focal entity
    const primary = this.narrateFocal(view.focal, memory, budget);
    
    // 3. Add context — but only what changes the user's likely next action
    const context = this.narrateContext(view.context, budget.remaining);
    
    // 4. Attach any affordances the user should know about
    const affordances = this.narrateAffordances(view.focal.affordances, budget);
    
    // 5. If disambiguation is pending, build the audio handshake
    const disambiguation = view.focal.renderHint.disambiguationPending
      ? this.buildAudioHandshake(view.executionState)
      : null;

    return {
      segments: [primary, ...context, ...affordances].filter(Boolean),
      disambiguation,
      interruptible: budget.type !== 'critical',
      expectedResponses: this.buildGrammar(view.focal.affordances),
    };
  }

  private computeBudget(view: ProjectedView): UtteranceBudget {
    // Proactive nudge = say as little as possible, don't break flow
    if (view.proactiveOffers.length > 0) {
      return { type: 'nudge', maxWords: 12, remaining: 12 };
    }
    
    // Disambiguation = say enough to differentiate options, nothing more
    if (view.focal.renderHint.disambiguationPending) {
      return { type: 'disambiguation', maxWords: 40, remaining: 40 };
    }
    
    // Explicit user query = full response, but still bounded
    return { type: 'full', maxWords: 80, remaining: 80 };
  }
}
```

## Narrating an entity

This is where a `SemanticEntity` becomes a sentence. The narration strategy depends on the entity's role, its relationship to the focal context, and what the user already knows (tracked via `InteractionMemory`).

```typescript
private narrateFocal(
  entity: ProjectedEntity,
  memory: InteractionMemory,
  budget: UtteranceBudget
): VoiceSegment {

  const alreadyKnows = memory.wasRecentlyNarrated(entity.id);

  switch (entity.role) {

    case 'source_file':
      // First time: full context. Subsequent: just what changed.
      return alreadyKnows
        ? { text: `${entity.name} — ${this.describeChange(entity)}`, priority: 1 }
        : { text: `You're in ${entity.name}, a ${this.describeFile(entity)}.`, priority: 1 };

    case 'diagnostic':
      // Always describe errors fully — they're high-salience regardless
      return {
        text: this.narrateError(entity),
        priority: 0,    // highest priority — speak first
        ssml: `<emphasis level="moderate">${this.narrateError(entity)}</emphasis>`
      };

    case 'function_symbol':
      return {
        text: `Function ${entity.name}: ${this.narrateSignature(entity)}`,
        priority: 1
      };

    case 'physical_device':
      // IoT entities are described by state, not structure
      return {
        text: `${entity.name} is currently ${this.describeDeviceState(entity)}.`,
        priority: 1
      };

    case 'task':
      return {
        text: `Task: ${entity.name}. Status: ${entity.attributes.status}.`,
        priority: 1
      };
  }
}

private narrateError(entity: ProjectedEntity): string {
  // Don't read the raw compiler error — synthesize it
  const message = entity.attributes.message;
  const line = entity.attributes.line;
  const severity = entity.attributes.severity;

  // "Type error on line 42: string is not assignable to number."
  return `${this.humanizeSeverity(severity)} on line ${line}: `
       + `${this.humanizeMessage(message)}`;
}

private humanizeMessage(raw: string): string {
  // The LLM does this — raw compiler errors are not speakable
  return this.llm.invoke(`
    Convert this compiler error to one clear spoken sentence, 
    no jargon, under 15 words: "${raw}"
  `);
}
```

The `humanizeMessage` call is important. Raw error strings like `Type 'string | undefined' is not assignable to type 'string'` are readable but not *speakable*. The voice renderer makes a lightweight LLM call to translate them before they hit TTS.

## The `VoiceScript` and SSML output

The assembled segments get serialized into SSML (Speech Synthesis Markup Language), which gives precise control over prosody — pauses, emphasis, rate — not just content.

```typescript
// packages/runtime/src/voice/ssml-builder.ts

export class SSMLBuilder {

  build(script: VoiceScript): string {
    const segments = script.segments
      .sort((a, b) => a.priority - b.priority)
      .map(s => this.segmentToSSML(s));

    return `
      <speak>
        ${segments.join('\n')}
        ${script.disambiguation
          ? this.buildDisambiguationSSML(script.disambiguation)
          : ''}
      </speak>
    `;
  }

  private segmentToSSML(segment: VoiceSegment): string {
    // Use pre-built SSML if the narrator provided it, otherwise wrap plaintext
    if (segment.ssml) return segment.ssml;

    return `
      <s>${segment.text}</s>
      <break time="200ms"/>
    `;
  }

  private buildDisambiguationSSML(d: AudioHandshake): string {
    // Read each option with a short pause between them
    // Use SSML's say-as for option labels so "A" isn't read as "uh"
    const options = d.options.map((opt, i) => `
      <s>
        <say-as interpret-as="characters">${PHONETIC_LABELS[i]}</say-as>:
        ${opt.description}
      </s>
      <break time="400ms"/>
    `).join('');

    return `
      <s>${d.prompt}</s>
      <break time="300ms"/>
      ${options}
      <s>Which would you like?</s>
    `;
  }
}
```

A concrete example of what this produces for an error disambiguation:

```xml
<speak>
  <s>Two matches for <emphasis>error handler</emphasis>.</s>
  <break time="300ms"/>
  <s><say-as interpret-as="characters">A</say-as>: 
     the catch block in auth dot t s, line 84</s>
  <break time="400ms"/>
  <s><say-as interpret-as="characters">B</say-as>: 
     the global error middleware in app dot t s, line 12</s>
  <break time="400ms"/>
  <s>Which would you like?</s>
</speak>
```

## Handling the response

The `VoiceScript` also carries an `expectedResponses` grammar — a structured set of what the user is likely to say next. This feeds the STT (speech-to-text) model to bias recognition toward valid responses rather than running open-domain transcription.

```typescript
export interface VoiceScript {
  segments: VoiceSegment[];
  disambiguation: AudioHandshake | null;
  interruptible: boolean;
  
  // Constrained grammar for the next expected utterance
  expectedResponses: ResponseGrammar;
}

export interface ResponseGrammar {
  type: 'open' | 'constrained' | 'confirmation';
  
  // For constrained: the set of valid responses and their mapped actions
  phrases?: Array<{
    utterances: string[];   // e.g. ["Alpha", "A", "the first one", "auth.ts"]
    action: string;         // e.g. "select_candidate_0"
  }>;
  
  // For confirmation: just yes/no
  confirmationPhrases?: { yes: string[], no: string[] };
}
```

When disambiguation is pending, the grammar is `constrained` and the STT model is given the candidate entity names as hints. *"Alpha"*, *"the auth one"*, *"the first"*, and *"auth dot t s"* all resolve to the same action. The user doesn't need to use the exact phonetic label.

## The interruptibility contract

The `interruptible: boolean` flag on `VoiceScript` governs a critical UX behaviour. If the user speaks while the TTS is playing, the runtime has to decide: stop and listen, or finish the sentence first?

```typescript
// packages/runtime/src/voice/tts-controller.ts

export class TTSController {
  
  async speak(script: VoiceScript): Promise<void> {
    const utterance = this.ssmlBuilder.build(script);
    
    // Start speaking
    const playback = this.ttsEngine.speak(utterance);
    
    if (script.interruptible) {
      // Barge-in: user can speak at any time to interrupt
      // The VAD (voice activity detector) signals us
      this.vad.onSpeechDetected(() => {
        playback.stop();
        this.audioCapture.startListening(script.expectedResponses);
      });
    } else {
      // Critical utterances (disambiguation options) play to completion
      // before the microphone opens — prevents partial captures
      await playback.finished();
      this.audioCapture.startListening(script.expectedResponses);
    }
  }
}
```

Proactive nudges are always interruptible — they should never hold the user hostage. Disambiguation option lists are non-interruptible — if the user speaks before hearing both options, you get a bad capture. This distinction is what separates a good voice UX from an annoying one.

## The full pipeline end-to-end

```
User speaks: "explain that error"
      │
      ▼
STT → IntentEnvelope { type: 'intent.information.explain',
                        target: CognitiveReference('that error') }
      │
      ▼
InteractionMemory resolves 'that error'
→ entity: diagnostic://ts-2345#L84
      │
      ▼
TopologicalExecutor: calls explainEntity(diagnostic://ts-2345#L84)
→ LLM produces explanation text
      │
      ▼
ProjectionEngine builds ProjectedView
(focal = the diagnostic entity, salience = 1.0)
      │
      ▼
VoiceProjector reshapes to VoiceScript
(humanizes the error, budgets to ~60 words,
 sets grammar to 'open' — follow-up questions expected)
      │
      ▼
SSMLBuilder serializes to marked-up speech
      │
      ▼
TTSController speaks, VAD listening for barge-in
      │
      ▼
User responds: "how do I fix it?"
→ new IntentEnvelope, cycle repeats
```

The key insight across this whole pipeline is that **no single component knows it's doing voice**. The `TopologicalExecutor` doesn't know. The `ProjectionEngine` doesn't know. Even the `VoiceProjector` doesn't know which TTS engine will consume its output — it produces SSML, a standard, and the `TTSController` handles the engine-specific integration. You could swap the entire TTS layer from a cloud API to an on-device model without touching anything upstream.
