# Message 12 — USIR

_Sent at: 2026-06-01 01:11:14_


 The A2U (Agent-to-USIR) protocol is where the agentic delegation model either earns trust or destroys it. Every other part of USIR optimises for speed and frictionlessness. The A2U protocol optimises for something harder: **keeping the human meaningfully in control of a process they aren't watching**.

---

## The fundamental tension

When you delegate a multi-hour task to a Worker Agent, you face a paradox. If the agent communicates too little, you have no idea what it's doing and can't catch mistakes early. If it communicates too much, every micro-decision becomes an interruption and delegation becomes pointless — you're just doing the work yourself via a slow proxy.

The A2U protocol resolves this by treating agent communication not as notifications, but as **intent proposals** flowing through the same semantic pipeline the user's own actions use. The agent doesn't "send messages" — it generates `IntentEnvelopes`, just like a human user would. The difference is who authorises them and when.

---

## The protocol structure

```typescript
// packages/protocol/src/a2u/index.ts

export interface A2UEnvelope {
  // The intent the agent wants to execute
  intent: IntentEnvelope;
  
  // What the agent knows about its own state
  agentState: AgentState;
  
  // Why the agent is surfacing this now
  surfacingReason: SurfacingReason;
  
  // How urgent this is — governs interruption behaviour
  urgency: 'background' | 'checkpoint' | 'blocker';
  
  // What happens if the user doesn't respond
  defaultBehaviour: DefaultBehaviour;
}

export interface AgentState {
  workerId: string;
  parentDelegateIntentId: string;   // The DelegateIntent that spawned this agent
  planProgress: PlanProgress;       // How far through the overall plan
  sandboxEntityIds: string[];        // What the agent can currently see/touch
  
  // Confidence in this specific step
  confidence: number;               // 0.0 – 1.0
  
  // What the agent is uncertain about, if anything
  uncertainty?: string;
}

export interface PlanProgress {
  totalSteps: number;
  completedSteps: number;
  currentPhase: string;             // Human-readable phase name
  estimatedRemaining: number;       // Milliseconds
}

export type SurfacingReason =
  | { type: 'checkpoint'; description: string }
    // Agent completed a logical phase and wants approval before continuing

  | { type: 'uncertainty'; question: string; options?: string[] }
    // Agent hit an ambiguity it cannot resolve from the sub-graph alone

  | { type: 'constraint-violation'; constraint: string; proposed: string }
    // Agent's next action would violate a declared constraint
    // but the agent has a proposed workaround

  | { type: 'completion'; summary: string }
    // Agent finished everything

  | { type: 'failure'; reason: string; recoverable: boolean };
    // Agent cannot proceed

export interface DefaultBehaviour {
  // If the user never responds, what should happen?
  action: 'proceed' | 'pause' | 'abort';
  
  // How long to wait before applying the default
  timeoutMs: number;
  
  // Only applies when action is 'proceed'
  proceedCondition?: string;   // e.g. "only if confidence > 0.9"
}
```

## The three trust tiers

Not all agent actions are equal. The A2U protocol separates them into three tiers based on how much the action changes the world and how reversible it is:

```typescript
// packages/runtime/src/a2u/trust-classifier.ts

export class TrustClassifier {

  classify(intent: IntentEnvelope): TrustTier {

    switch (intent.intent.type) {

      // Tier 1 — read-only: agent can do this without asking
      case 'intent.information.explain':
      case 'intent.information.compare':
      case 'intent.navigation.locate':
        return {
          tier: 1,
          requiresApproval: false,
          logToProvenance: true,
        };

      // Tier 2 — reversible mutation: agent should checkpoint,
      // but can proceed with confidence above threshold
      case 'intent.manipulation.edit':
      case 'intent.manipulation.create':
        return {
          tier: 2,
          requiresApproval: intent.agentConfidence < 0.85,
          logToProvenance: true,
          reversible: true,
        };

      // Tier 3 — irreversible or high-impact: always requires approval
      case 'intent.manipulation.delete':
      case 'intent.execution.run':        // Running code
      case 'intent.collaboration.share':  // Sending to external parties
        return {
          tier: 3,
          requiresApproval: true,
          logToProvenance: true,
          reversible: false,
        };
    }
  }
}
```

The key insight here is that `requiresApproval` is a function of both the intent *type* and the agent's *confidence*. A high-confidence edit (`confidence: 0.94`) can proceed automatically. The same edit at low confidence (`confidence: 0.61`) gets surfaced as a checkpoint. The user set the `0.85` threshold when they issued the `DelegateIntent` — it lives in `constraints`.

## The `A2UDispatcher`

When the agent generates an `A2UEnvelope`, it goes to the `A2UDispatcher` on the main USIR runtime. This is the gatekeeper that decides what to do with it:

```typescript
// packages/runtime/src/a2u/dispatcher.ts

export class A2UDispatcher {

  async dispatch(envelope: A2UEnvelope): Promise<DispatchResult> {

    const trust = this.trustClassifier.classify(envelope.intent);

    // 1. Always record to provenance — even background actions
    await this.provenanceStore.record({
      intentId: envelope.intent.intentId,
      actor: {
        type: 'agent',
        id: envelope.agentState.workerId,
        confidence: envelope.agentState.confidence,
      },
      rationale: {
        type: 'delegated',
        planStep: envelope.agentState.planProgress.currentPhase,
        interpretedIntent: envelope.intent.intent.instruction,
      },
      authorization: {
        type: trust.requiresApproval ? 'pending' : 'delegated',
        delegateIntentId: envelope.agentState.parentDelegateIntentId,
        allowedEntityIds: envelope.agentState.sandboxEntityIds,
        signature: await this.sign(envelope),
      },
      causalParents: [envelope.agentState.parentDelegateIntentId],
      timestamp: Date.now(),
      contentHash: await this.hashTarget(envelope.intent),
    });

    // 2. Route based on trust tier and urgency
    if (!trust.requiresApproval) {
      return this.executeImmediate(envelope, trust);
    }

    if (envelope.urgency === 'background') {
      return this.queueForReview(envelope);
    }

    if (envelope.urgency === 'checkpoint') {
      return this.surfaceCheckpoint(envelope);
    }

    if (envelope.urgency === 'blocker') {
      return this.interruptUser(envelope);
    }
  }

  private async executeImmediate(
    envelope: A2UEnvelope,
    trust: TrustTier
  ): Promise<DispatchResult> {
    // Execute directly — agent had sufficient confidence
    // and trust tier allows it
    const result = await this.topologicalExecutor.execute(envelope.intent);
    
    // Notify the agent of the outcome
    await this.agentBus.send(envelope.agentState.workerId, {
      type: 'execution-result',
      intentId: envelope.intent.intentId,
      success: result.success,
      output: result.output,
    });

    return { status: 'executed', result };
  }

  private async queueForReview(
    envelope: A2UEnvelope
  ): Promise<DispatchResult> {
    // Add to the review queue — user will see it at their next natural pause
    // Don't interrupt ongoing work
    await this.reviewQueue.enqueue(envelope);
    
    // Schedule a gentle nudge for the next idle moment
    this.idleScheduler.scheduleNudge({
      waypointBuilder: () => this.buildReviewWaypoint(envelope),
      priority: 'low',
    });

    return { status: 'queued' };
  }

  private async surfaceCheckpoint(
    envelope: A2UEnvelope
  ): Promise<DispatchResult> {
    // Build a waypoint and push to all active surfaces
    const waypoint = this.buildCheckpointWaypoint(envelope);
    await this.surfaceRouter.push(waypoint);
    
    // Apply default behaviour after timeout if user doesn't respond
    this.timeoutManager.set(
      envelope.intent.intentId,
      envelope.defaultBehaviour.timeoutMs,
      () => this.applyDefault(envelope)
    );

    return { status: 'pending-approval', waypointId: waypoint.id };
  }

  private async interruptUser(
    envelope: A2UEnvelope
  ): Promise<DispatchResult> {
    // Blocker — agent cannot proceed without input
    // This overrides the user's current focus
    const waypoint = this.buildBlockerWaypoint(envelope);

    // Push to all surfaces simultaneously — watch haptic fires,
    // screen shows overlay, XR shows urgent spatial indicator
    await this.surfaceRouter.pushUrgent(waypoint);

    // No timeout default — blockers require explicit resolution
    return { status: 'blocked', waypointId: waypoint.id };
  }
}
```

## Building the review waypoint

When the agent surfaces a checkpoint, the waypoint it generates needs to carry enough context for the user to make a decision without having to re-read the whole codebase. This is the hardest UX problem in agentic systems — context compression for human review.

```typescript
private buildCheckpointWaypoint(envelope: A2UEnvelope): InteractionWaypoint {

  const { intent, agentState, surfacingReason } = envelope;

  // Build a diff summary — what changed, expressed semantically
  const diff = this.diffSummarizer.summarize(
    intent,
    agentState.sandboxEntityIds
  );

  return {
    id: `checkpoint-${envelope.intent.intentId}`,
    context: {
      state: 'agent-checkpoint',
      objective: `Review: ${agentState.planProgress.currentPhase}`,
    },

    presentations: {

      display: {
        layout: 'diff-review',
        header: {
          title: agentState.planProgress.currentPhase,
          subtitle: `Step ${agentState.planProgress.completedSteps + 1} `
                  + `of ${agentState.planProgress.totalSteps}`,
          progressFraction: agentState.planProgress.completedSteps
                          / agentState.planProgress.totalSteps,
        },
        // The diff — what files changed, what the changes mean
        body: diff.entries.map(entry => ({
          entityName: entry.entityName,
          summary: entry.humanSummary,      // "Added audience validation to JWT check"
          linesChanged: entry.linesChanged,
          reversible: entry.reversible,
        })),
        // Confidence signal — helps user calibrate how much to scrutinise
        confidence: {
          value: agentState.confidence,
          label: this.confidenceLabel(agentState.confidence),
          // "High confidence" / "Some uncertainty" / "Review carefully"
        },
        primaryAction:   { label: 'Approve', action: 'approve' },
        secondaryAction: { label: 'Reject',  action: 'reject' },
        tertiaryAction:  { label: 'Discuss', action: 'discuss' },
      },

      audio: {
        // Spoken summary of the checkpoint — what changed and why
        tts: this.buildCheckpointUtterance(envelope, diff),
        ssml: this.buildCheckpointSSML(envelope, diff),
        autoPlay: false,   // User initiates — they might be in a meeting
      },

      haptic: {
        // Double tap — "needs your decision"
        pattern: 'notification-double',
        timing: 'immediate',
        repeat: false,
      },

      // XR: show a spatial panel near the files that changed
      spatial: {
        anchorToEntityId: diff.primaryEntityId,
        layout: 'floating-panel',
        content: diff.entries[0]?.humanSummary ?? '',
      },
    },

    expectedInputs: {
      voice: {
        grammar: {
          type: 'constrained',
          phrases: [
            { utterances: ['approve', 'looks good', 'yes', 'proceed'], action: 'approve' },
            { utterances: ['reject', 'no', 'undo', 'revert'],          action: 'reject' },
            { utterances: ['discuss', 'explain', 'why', 'tell me more'], action: 'discuss' },
          ],
        },
      },
      touch: {
        events: [
          { target: 'approve-button', action: 'approve' },
          { target: 'reject-button',  action: 'reject' },
          { target: 'discuss-button', action: 'discuss' },
        ],
      },
    },
  };
}

private buildCheckpointUtterance(
  envelope: A2UEnvelope,
  diff: DiffSummary
): string {
  const { agentState } = envelope;
  const phase = agentState.planProgress.currentPhase;
  const filesChanged = diff.entries.length;
  const confidence = this.confidenceLabel(agentState.confidence);

  // "Migration phase one complete. Three files changed.
  //  Confidence: high. Main change: added audience validation to the JWT check.
  //  Approve, reject, or discuss?"
  return `${phase} complete. ${filesChanged} `
       + `${filesChanged === 1 ? 'file' : 'files'} changed. `
       + `Confidence: ${confidence}. `
       + `Main change: ${diff.entries[0]?.humanSummary ?? 'see display'}. `
       + `Approve, reject, or discuss?`;
}
```

## The three resolution paths

When the user responds to a checkpoint, there are three outcomes. Each feeds back into the agent differently.

```typescript
// packages/runtime/src/a2u/resolver.ts

export class A2UResolver {

  async resolve(
    waypointId: string,
    action: 'approve' | 'reject' | 'discuss',
    envelope: A2UEnvelope
  ): Promise<void> {

    switch (action) {

      case 'approve': {
        // Record approval in provenance
        await this.provenanceStore.update(envelope.intent.intentId, {
          authorization: { type: 'approved', approvalIntentId: this.currentIntentId() }
        });

        // Execute the approved intent
        const result = await this.topologicalExecutor.execute(envelope.intent);

        // Notify agent — it's waiting for this signal to continue
        await this.agentBus.send(envelope.agentState.workerId, {
          type: 'checkpoint-approved',
          intentId: envelope.intent.intentId,
          result,
        });
        break;
      }

      case 'reject': {
        // Record rejection
        await this.provenanceStore.update(envelope.intent.intentId, {
          authorization: { type: 'rejected' }
        });

        // Notify agent — it needs to revise its plan
        await this.agentBus.send(envelope.agentState.workerId, {
          type: 'checkpoint-rejected',
          intentId: envelope.intent.intentId,
          // Agent receives no explanation by default — user can add one
          // via the 'discuss' path if they want the agent to understand why
        });
        break;
      }

      case 'discuss': {
        // Open a synchronous dialogue loop with the agent
        // This is an L3 ExplainIntent directed at the agent, not an entity
        await this.openAgentDialogue(envelope);
        break;
      }
    }
  }

  private async openAgentDialogue(envelope: A2UEnvelope): Promise<void> {
    // Switch the current intent context to a dialogue with the agent
    // The agent's responses flow back through the A2U pipeline
    // but with surfacingReason.type: 'uncertainty' to signal it's a dialogue turn

    // The user's USIR instance now acts as a relay:
    // user voice/text → IntentEnvelope → agentBus → agent
    // agent response → A2UEnvelope → back to user surface

    this.dialogueManager.openChannel({
      agentId: envelope.agentState.workerId,
      contextEntityIds: this.extractContextEntities(envelope),
      // The diff that triggered the discussion becomes the conversation context
      seedContext: this.buildDialogueContext(envelope),
    });
  }
}
```

## The uncertainty case — agent asks a question

The checkpoint path covers planned communication. But agents also hit unexpected ambiguities mid-execution. When the agent encounters something its sub-graph projection doesn't resolve, it surfaces an `uncertainty` envelope — and this one cannot have a default behaviour of `proceed`:

```typescript
// Inside the Worker Agent
async function handleAmbiguity(
  question: string,
  options: string[],
  blockedIntentId: string
): Promise<string> {

  // Pause execution and surface the question
  await a2uChannel.send({
    intent: createBlockedIntent(blockedIntentId),
    agentState: this.currentState(),
    surfacingReason: {
      type: 'uncertainty',
      question,
      options,
      // "The users table has two email columns: email and contact_email.
      //  Which should I use as the DynamoDB partition key?"
    },
    urgency: 'blocker',
    defaultBehaviour: {
      action: 'pause',   // Cannot guess here — must wait
      timeoutMs: 0,      // No timeout — this is a genuine blocker
    },
  });

  // Block until the user resolves
  return new Promise(resolve => {
    this.pendingResolutions.set(blockedIntentId, resolve);
  });
}
```

The runtime receives this, builds a waypoint with `expectedInputs` matching the agent's provided options, and interrupts the user. When the user responds — by voice, tap, or crown selection on the watch — the resolution flows back through the `agentBus` and the agent's `Promise` resolves. Execution resumes from exactly where it paused, with the user's answer injected as a resolved value in the plan.

## The constraint violation case

This one is subtle. The agent has already violated a declared constraint — *"do not mutate production config"* — but it has a proposed workaround and is asking for explicit permission rather than silently doing the wrong thing:

```typescript
await a2uChannel.send({
  intent: proposedWorkaroundIntent,
  agentState: this.currentState(),
  surfacingReason: {
    type: 'constraint-violation',
    constraint: 'Do not mutate production config',
    proposed: 'Create a new staging config file instead, then ask you to promote it',
    // The agent is not asking "can I break the rule?"
    // It is asking "can I do this different thing that achieves the goal?"
  },
  urgency: 'checkpoint',
  defaultBehaviour: {
    action: 'pause',
    timeoutMs: 300_000,   // 5 minutes — then pause and wait indefinitely
  },
});
```

The constraint violation path is the A2U protocol's most important safety feature. An agent that silently works around constraints is far more dangerous than one that stops and asks. The protocol makes the right behaviour — surfacing violations explicitly — the path of least resistance for the agent implementation.

## The completion envelope

When the agent finishes, the final `A2UEnvelope` carries a full summary of everything it did, structured so the user can verify the work matches what they originally asked for:

```typescript
await a2uChannel.send({
  intent: createCompletionMarker(this.parentDelegateIntentId),
  agentState: {
    ...this.currentState(),
    planProgress: {
      ...this.planProgress,
      completedSteps: this.planProgress.totalSteps,
      estimatedRemaining: 0,
    },
  },
  surfacingReason: {
    type: 'completion',
    summary: await this.buildCompletionSummary(),
    // Summary is generated by the agent's LLM from the full provenance log
    // of everything it did — not hand-written
  },
  urgency: 'checkpoint',
  defaultBehaviour: {
    action: 'proceed',   // "Proceeding" here just means "mark task done"
    timeoutMs: 86_400_000,  // 24 hours — no rush
  },
});
```

The `buildCompletionSummary` is itself a lightweight LLM call. The agent reads its own provenance log — every `A2UEnvelope` it sent, every approval and rejection it received — and synthesises a plain-language summary: *"Migrated auth schema to DynamoDB. Created three new files, modified two existing ones. Skipped the billing table after you rejected that step. All tests pass."*

## The invariant the whole protocol rests on

The A2U protocol works because of one rule that cannot be violated: **the agent never mutates state without an entry in the provenance log, and the provenance log is the source of truth for what happened and why**.

Every `executeImmediate`, every `checkpoint-approved`, every `constraint-violation` that got a yes — all of it is in the log with a timestamp, a confidence score, and a causal link to the `DelegateIntent` that authorised the entire operation. If anything goes wrong, the user can walk the provenance graph and see exactly which decision led to the problem, whether it was a human approval or an automatic execution, and what the agent's stated confidence was at that moment.

Trust in autonomous agents isn't built by making them infallible. It's built by making their reasoning transparent enough that you can catch mistakes early, understand why they happened, and fix them precisely — without having to throw away everything they did.
