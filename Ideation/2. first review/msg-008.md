 This is the architecture where the ambient computing vision becomes most concrete. XR glasses are the purest expression of the thin-client model — they have almost no processing headroom, they're always on, and they're physically attached to the user's body. That combination of constraints forces every architectural decision to be exactly right.

---

## What "thin" actually means here

A screen client (VS Code extension, browser) runs on a machine with gigabytes of RAM and a fast GPU. It can afford to do some local computation — debouncing, layout calculation, animation. XR glasses in 2024–2026 have the thermal budget of a smartphone and the form factor of eyewear. They cannot run the projection engine. They cannot run the LLM. They cannot maintain the semantic graph.

What they *can* do:

- Capture a gaze vector at 120Hz
- Capture audio continuously
- Render lightweight 2D panels and simple 3D anchors
- Send sensor data over a low-latency local connection (BLE, WiFi 6, UWB)
- Receive and display a `ProjectedView` payload

That's the entire contract. Everything else lives on the phone or home server.

The glasses never talk to the USIR runtime directly. They talk to the `SpatialAdapter` running on the phone, which does the heavy sensor processing and produces clean semantic events. This keeps the glasses-to-phone link as a sensor stream, not a compute offload.

## The `XRProjector`

Just as the voice path has a `VoiceProjector` that reshapes `ProjectedView` for linear audio, the XR path has an `XRProjector` that reshapes it for spatial placement. The core problem it solves: *where in 3D space does each piece of information live?*

```typescript
// packages/runtime/src/projection/xr-projector.ts

export class XRProjector {

  project(view: ProjectedView, spatialContext: SpatialContext): XRScene {
    
    return {
      panels: this.placePanels(view, spatialContext),
      anchors: this.placeAnchors(view, spatialContext),
      highlights: this.computeHighlights(view, spatialContext),
      ambientIndicators: this.computeAmbientIndicators(view),
    };
  }

  private placePanels(
    view: ProjectedView,
    ctx: SpatialContext
  ): SpatialPanel[] {

    const panels: SpatialPanel[] = [];

    // Primary focal panel — always near gaze, never occluding real world
    panels.push({
      id: 'focal',
      content: this.renderEntityContent(view.focal),
      placement: {
        strategy: 'gaze-anchored',
        // Offset slightly below and right of gaze center
        // so it doesn't sit directly over what the user is looking at
        gazeOffset: { x: 0.15, y: -0.12 },
        distance: 0.6,         // metres from eye plane
        maxWidth: 0.3,         // metres, ~A5 page at arm's length
      },
      opacity: view.focal.renderHint.salience,
      interactable: true,
    });

    // Context panels — positioned relative to physical space, not gaze
    view.context
      .filter(e => e.renderHint.salience > 0.3)
      .forEach((entity, i) => {
        const anchor = ctx.findSurfaceAnchor(entity.id)
          ?? ctx.peripheralSlot(i);

        panels.push({
          id: entity.id,
          content: this.renderEntityContent(entity),
          placement: {
            strategy: anchor ? 'surface-anchored' : 'peripheral',
            anchor,
            peripheralAngle: i * 45,   // degrees from gaze center
            distance: anchor ? 0 : 1.2,
          },
          opacity: entity.renderHint.salience * 0.7,
          interactable: false,
        });
      });

    return panels;
  }

  private placeAnchors(
    view: ProjectedView,
    ctx: SpatialContext
  ): SpatialAnchor[] {
    // Anchors are minimal visual tags on physical objects
    // e.g. a small glowing ring around the lamp entity
    return view.context
      .filter(e => e.role === 'physical_device' || e.role === 'spatial_anchor')
      .map(e => ({
        entityId: e.id,
        physicalVolume: ctx.getVolume(e.id),
        style: e.renderHint.disambiguationPending
          ? 'pulsing-ring'      // disambiguation pending — draw attention
          : 'subtle-outline',   // normal state — just acknowledge presence
        label: e.name,
        affordanceHint: e.renderHint.primaryAffordance,
      }));
  }

  private computeHighlights(
    view: ProjectedView,
    ctx: SpatialContext
  ): SpatialHighlight[] {
    // During disambiguation: paint phonetic labels on physical objects
    if (!view.focal.renderHint.disambiguationPending) return [];

    return view.executionState?.candidates.map((candidate, i) => ({
      entityId: candidate.entityId,
      label: PHONETIC_LABELS[i],
      style: 'disambiguation-overlay',
      physicalVolume: ctx.getVolume(candidate.entityId),
    })) ?? [];
  }

  private computeAmbientIndicators(view: ProjectedView): AmbientIndicator[] {
    // Proactive offers render as subtle peripheral glow, not a panel
    return view.proactiveOffers.map(offer => ({
      type: 'peripheral-glow',
      targetEntityId: offer.entityId,
      color: 'soft-amber',
      // Pulses slowly to draw attention without demanding it
      animation: 'slow-pulse',
      // Disappears if user's gaze doesn't move toward it within 3 seconds
      autoDecay: 3000,
    }));
  }
}
```

## The `SpatialContext` — knowing where things are

The `XRProjector` needs to know the physical location of entities to place panels and anchors correctly. This comes from the `SpatialContext`, maintained by the `SpatialAdapter` on the phone from the room mesh and UWB tracking data:

```typescript
export interface SpatialContext {
  // Current room mesh — updated as the glasses scan
  roomMesh: SpatialVolume[];
  
  // Known physical entity locations
  entityVolumes: Map<string, SpatialVolume>;
  
  // Available "surface anchors" — flat surfaces where panels can attach
  surfaces: Surface[];
  
  // Current head pose — where the user is looking and facing
  headPose: HeadPose;
  
  // Available peripheral slots — positions in the user's visual field
  // that don't currently contain real-world objects worth looking at
  peripheralSlots: PeripheralSlot[];
  
  findSurfaceAnchor(entityId: string): Surface | null;
  getVolume(entityId: string): SpatialVolume | null;
  peripheralSlot(index: number): PeripheralSlot;
}
```

The `findSurfaceAnchor` method is key. When a digital entity has a corresponding physical object — a document on a desk, a smart device on a wall — the context panel for that entity attaches to the surface rather than floating in gaze space. This is what makes XR feel *grounded* rather than like a floating HUD.

## The render client on the glasses

The glasses-side `RenderClient` is intentionally minimal. It receives an `XRScene` payload and maps it to the glasses' native rendering API. It has no logic beyond "render this, here, at this opacity":

```typescript
// apps/xr-client/src/render-client.ts

export class XRRenderClient {
  
  private activeScene: XRScene | null = null;

  onSceneUpdate(scene: XRScene) {
    this.reconcile(this.activeScene, scene);
    this.activeScene = scene;
  }

  private reconcile(prev: XRScene | null, next: XRScene) {
    // Panels: add, update, remove
    const prevIds = new Set(prev?.panels.map(p => p.id) ?? []);
    const nextIds = new Set(next.panels.map(p => p.id));

    // Remove panels no longer in scene
    for (const id of prevIds) {
      if (!nextIds.has(id)) this.nativeRenderer.removePanel(id);
    }

    // Add or update panels
    for (const panel of next.panels) {
      if (!prevIds.has(panel.id)) {
        this.nativeRenderer.createPanel(panel);
      } else {
        this.nativeRenderer.updatePanel(panel);
      }
    }

    // Anchors and highlights follow the same reconciliation pattern
    this.reconcileAnchors(prev?.anchors ?? [], next.anchors);
    this.reconcileHighlights(prev?.highlights ?? [], next.highlights);
    this.reconcileAmbientIndicators(prev?.ambientIndicators ?? [], next.ambientIndicators);
  }
}
```

The reconciler is the most complex thing the glasses run. Everything else is the native renderer doing panel placement using the `SpatialPanel` coordinates provided by the phone. The glasses are doing layout, not computation.

## The sensor bridge — closing the loop

The glasses aren't just a display. They're also the primary input device. The `SensorBridge` packages raw sensor data and sends it to the phone's `SpatialAdapter`:

```typescript
// apps/xr-client/src/sensor-bridge.ts

export class SensorBridge {
  
  private lastGazeEntityId: string | null = null;

  start() {
    // Eye tracking at 120Hz — but we only send semantic events, not raw vectors
    this.eyeTracker.onFrame(gaze => {
      // The raycast runs locally on the glasses — it's just math, not AI
      const hitEntity = this.raycast(gaze, this.localEntityVolumes);
      
      if (hitEntity?.id !== this.lastGazeEntityId) {
        // Only send when gaze crosses a semantic boundary
        // and the user has dwelled for >200ms (filters out saccades)
        if (this.dwellTimer.hasDwelled(200)) {
          this.send({
            type: 'gaze_entity_change',
            entityId: hitEntity?.id ?? null,
            headPose: this.imu.currentPose(),
          });
          this.lastGazeEntityId = hitEntity?.id ?? null;
        }
      }
    });

    // Audio — continuous capture, VAD on-device to avoid streaming silence
    this.microphone.onSpeechDetected(audio => {
      this.send({ type: 'audio_chunk', data: audio });
    });

    // Room mesh — send deltas only when geometry changes significantly
    this.cameras.onMeshDelta(delta => {
      if (delta.significance > MESH_UPDATE_THRESHOLD) {
        this.send({ type: 'mesh_delta', delta });
      }
    });
  }
  
  private raycast(
    gaze: GazeVector,
    volumes: SpatialVolume[]
  ): SpatialVolume | null {
    // Pure math — finds which volume the gaze ray intersects first
    // This is the only compute that justifiably runs on the glasses
    // because it needs to happen at 120Hz with zero latency
    return volumes
      .filter(v => intersectsRay(gaze, v))
      .sort((a, b) => distanceAlongRay(gaze, a) - distanceAlongRay(gaze, b))[0]
      ?? null;
  }
}
```

The most important line in the whole client is the gaze filter: `if (hitEntity?.id !== this.lastGazeEntityId)`. The 120Hz eye tracking stream gets compressed down to a trickle of semantic boundary-crossing events. The phone's `SpatialAdapter` receives maybe 2–5 gaze events per second under normal use, not 120 frames per second of raw vectors. This is what keeps the glasses' radio from becoming the bottleneck.

## What this enables that screens cannot

The architectural payoff shows up in three scenarios that are simply impossible with screen-bound software.

When the user **looks at a physical object and speaks** — *"dim that to fifty percent"* — the gaze event and the audio arrive at the `SpatialAdapter` within milliseconds of each other. The `FusedIntent` is constructed with `target: iot://philips-hue/lamp-3` resolved from the raycast, and the runtime executes `intent.manipulation.edit` on the lamp entity. No app switching. No menu navigation. The physical world is directly addressable.

When the user is **pair-programming via L8 Collaboration** — peer's `DiscussIntent` arrives targeting `file://auth.ts#L84` — the XR glasses can render the peer's "presence" as a spatial indicator floating near wherever the user last touched that file, even if the file isn't currently visible. The digital artefact has a physical address in the room.

When the **proactive engine fires** — detecting a stuck loop on a type error — the XR client renders it as a peripheral amber glow near the relevant anchor, not as a notification panel demanding attention. The user can glance at it or ignore it. It decays in three seconds either way. This is only possible because the proactive offer is expressed as an `AmbientIndicator` in the scene, not as a UI component — the XR renderer chose the right affordance for the medium automatically.

## The invariant that holds it together

At every layer of this pipeline — `SensorBridge`, `SpatialAdapter`, `XRProjector`, `RenderClient` — the same rule applies: **process at the boundary, pass semantics inward**. Raw gaze vectors never reach the runtime. Raw audio never reaches the intent router without VAD filtering. Raw mesh data never reaches the projection engine without delta compression. Each layer does exactly the filtering its position in the pipeline justifies, and nothing more.

The glasses are thin not because we made them thin — but because every layer between the sensor and the semantic core strips away the noise that layer is uniquely positioned to remove.
