 Great question — this is genuinely the missing piece that elevates USIR from a fast interaction runtime to a *trustworthy* one.

## The core idea

Every mutation in USIR today answers *what* changed and *when*. The Provenance layer (L0.5) answers *why* it changed, *who* authorized it, and *what reasoning* justified it. This is distinct from an undo log — it's a causal graph of decisions.

---

## The `ProvenanceEnvelope`

Every `IntentEnvelope` that mutates state gets a provenance record attached before execution:

```typescript
// packages/protocol/src/provenance/index.ts

export interface ProvenanceEnvelope {
  intentId: string;           // Links back to the IntentEnvelope
  
  // Who or what initiated this
  actor: ProvenanceActor;
  
  // Why — the reasoning chain
  rationale: Rationale;
  
  // What authorized this mutation
  authorization: Authorization;
  
  // Causal lineage — what prior state / prior intents led here
  causalParents: string[];    // intentIds this mutation depends on or responds to
  
  timestamp: number;
  contentHash: string;        // Hash of the mutated entity's state after execution
}

export interface ProvenanceActor {
  type: 'human' | 'agent' | 'proactive_engine' | 'peer';
  id: string;                 // DID or session ID
  modality?: string;          // 'voice', 'gaze', 'keyboard', 'api'
  confidence?: number;        // For proactive/agent actors — how certain was the trigger?
}

export interface Rationale {
  type: 'explicit' | 'inferred' | 'delegated' | 'proactive';
  
  // The human's original utterance or gesture, if any
  rawSignal?: string;         // e.g. "refactor this for readability"
  
  // The LLM's reasoning — what it understood
  interpretedIntent?: string; // e.g. "rename variables and extract helper functions"
  
  // For agents: the step in the execution plan
  planStep?: string;
  
  // For proactive: the implicit signal that triggered this
  implicitTrigger?: ImplicitTrigger;
}

export interface ImplicitTrigger {
  signal: 'dwell' | 'context_thrash' | 'error_fixation' | 'typing_halted';
  durationMs: number;
  entityId: string;           // What the user was stuck on
  confidence: number;
}

export interface Authorization {
  type: 'direct' | 'delegated' | 'approved' | 'pending';
  
  // For delegated: which DelegateIntent granted this agent permission
  delegateIntentId?: string;
  
  // For approved: the ConfirmIntent the human issued
  approvalIntentId?: string;
  
  // Capability scope — what this actor was allowed to touch
  allowedEntityIds: string[]; // Sub-graph projection at time of authorization
  
  // Cryptographic signature (decentralized, no central server)
  signature: string;
}
```

## The causal graph structure

Rather than a flat log, provenance forms a **DAG** — directed acyclic graph — where each mutation node points to its causal parents. This is what makes it powerful.

```
[Human: "migrate auth to DynamoDB"]   <── root intent
         │
         ▼
[DelegateIntent → Worker Agent A]     <── authorization boundary
         │
    ┌────┴─────┐
    ▼           ▼
[EditIntent:    [CreateIntent:         <── agent mutations, both children
 schema.ts]      migration.ts]              of the same DelegateIntent
    │
    ▼
[Human: "Approve"]                    <── approval closes the chain
```

If the human later asks *"why does schema.ts look like this?"*, the runtime can walk the causal graph and reconstruct: *"Worker Agent A edited this during the DynamoDB migration you approved on Tuesday, specifically because the PostgreSQL UUID type is incompatible with DynamoDB's String partition key."*

## Integration points in the existing architecture

**In the `TopologicalExecutor`**, before executing any plan step, it calls `ProvenanceStore.record()`:

```typescript
// Inside TopologicalExecutor.ts
async executeStep(step: PlanStep, plan: ExecutionPlan) {
  
  const provenance: ProvenanceEnvelope = {
    intentId: step.intentId,
    actor: {
      type: plan.actor,
      id: plan.actorId,
      modality: plan.modality,
      confidence: plan.confidence,
    },
    rationale: {
      type: plan.actor === 'agent' ? 'delegated' : 'explicit',
      rawSignal: plan.sourceUtterance,
      interpretedIntent: plan.llmSummary,
      planStep: step.description,
    },
    authorization: {
      type: plan.requiresApproval ? 'pending' : 'direct',
      delegateIntentId: plan.parentDelegateId,
      allowedEntityIds: plan.sandboxEntityIds,
      signature: await sign(step, this.identityKey),
    },
    causalParents: plan.dependsOn,
    timestamp: Date.now(),
    contentHash: await hashEntityState(step.targetId),
  };

  await this.provenanceStore.record(provenance);
  return this.executeHook(step);
}
```

**In the Proactive Engine**, the `ImplicitTrigger` gets recorded so the human can later understand *why* the system offered help unprompted — and the system can learn from whether those offers were accepted or dismissed.

**In the Disambiguation UI**, when a human picks option "Bravo" during Collaborative Narrowing, a `rationale.type: 'explicit'` record captures the original ambiguous utterance and the resolved choice side-by-side.

## What this unlocks

**Explainability on demand.** A new L3 intent: `ExplainMutationIntent` — *"why did this file change?"* The runtime walks the provenance graph and synthesizes a plain-language explanation from the stored rationale chain. No reconstruction required.

**Auditable agent delegation.** For enterprise contexts, the entire causal chain of a Worker Agent's actions can be exported as a signed, tamper-evident log. Each `ProvenanceEnvelope` is cryptographically linked to the `DelegateIntent` that authorized it.

**Adaptive proactive confidence.** The Proactive Engine already tracks whether users ignore its nudges. With provenance, this becomes richer: not just *"they ignored CSS suggestions"* but *"they ignored 6 dwell-triggered suggestions during focused coding sessions but accepted 4 context-thrash suggestions during debugging."* The confidence threshold becomes a nuanced per-signal-per-context model rather than a flat scalar.

**Semantic rollback, not just undo.** Standard undo reverses the last mutation. Semantic rollback can target a specific causal subtree — *"undo everything the migration agent touched, but keep the test scaffolding I added manually on top."* Because provenance tracks `causalParents`, the runtime can compute exactly which mutations are safe to revert without touching their dependents.

**Peer trust in federated collaboration.** When User B receives a `ShareIntent` from User A, the attached provenance tells them not just *what* the entity's current state is, but the full history of *who touched it and why*. This is the difference between receiving a file and receiving a file with a commit history.

---

## The one design constraint worth emphasizing

Provenance should be **write-once and append-only** at the storage layer. Once a `ProvenanceEnvelope` is committed, it cannot be edited — only superseded by a new envelope (e.g. a human overriding an agent decision creates a new node pointing back to the original, rather than erasing it). This is what makes the causal graph trustworthy, and it's the property that makes the cryptographic signatures meaningful.
