# Message 10 — USIR

_Sent at: 2026-06-01 01:09:08_


 The smartwatch is the most constrained surface in the USIR ecosystem — and that constraint is what makes it the most instructive. A screen has infinite layout flexibility. Voice has time as its canvas. XR has three-dimensional space. A smartwatch has a 44mm display, one speaker barely capable of voice playback, a vibration motor, and a user whose wrist is probably moving.

Getting USIR to work well here isn't a rendering problem. It's a **prioritization and translation problem**.

---

## What the watch can actually do

Before any architecture, the honest capability inventory:

```typescript
export interface WatchCapabilities {
  display: {
    available: true;
    size: 'small';           // ~184×224pt on a typical smartwatch
    interaction: 'tap' | 'digital-crown' | 'force-touch';
    // Cannot render dense text, multi-column layouts, or nested UI
  };
  audio: {
    available: boolean;      // Not all watches have speakers
    quality: 'low';          // Adequate for short TTS, not music
    input: boolean;          // Microphone — most watches have one
  };
  haptic: {
    available: true;
    patterns: HapticPattern[];  // A fixed library of named patterns
    // Cannot do arbitrary vibration — must use the OS's named set
  };
  compute: 'minimal';        // Enough to reconcile a diff, not to run logic
  network: 'proxied';        // All network goes through the paired phone
}
```

The `network: 'proxied'` constraint is important. The watch doesn't talk to the USIR runtime directly — it talks to the phone, which runs the full runtime and pushes `InteractionWaypoint` payloads down to the watch. This is the same thin-client split as XR glasses, but even more constrained on the rendering end.

## The `WatchProjector`

The watch needs its own projection stage, downstream of the main `ProjectionEngine`, that reshapes a `ProjectedView` into something expressible within watch constraints. Its primary job is radical reduction — taking a potentially rich semantic context and deciding what single thing matters most right now.

```typescript
// packages/runtime/src/projection/watch-projector.ts

export class WatchProjector {

  project(
    view: ProjectedView,
    caps: WatchCapabilities,
    context: WatchContext
  ): InteractionWaypoint {

    // 1. Find the single most important thing to express
    const focal = this.selectFocal(view, context);

    // 2. Build the waypoint — parallel branches for each modality
    return {
      id: `waypoint-${Date.now()}`,
      context: {
        state: view.meta.projectedAt,
        objective: focal.renderHint.primaryAffordance,
      },
      presentations: {
        display: caps.display.available
          ? this.buildDisplayPresentation(focal, view)
          : null,
        audio: caps.audio.available
          ? this.buildAudioPresentation(focal, view)
          : null,
        haptic: this.buildHapticPresentation(focal, view),
      },
      expectedInputs: this.buildExpectedInputs(focal, caps),
    };
  }

  private selectFocal(
    view: ProjectedView,
    context: WatchContext
  ): ProjectedEntity {

    // If the watch is being raised (wrist-raise gesture), 
    // the user wants an at-a-glance summary — pick highest salience
    if (context.gestureHint === 'wrist-raise') {
      return view.focal;
    }

    // If there's a pending proactive offer, that takes priority —
    // the haptic already fired, now we need to show what it was about
    if (view.proactiveOffers.length > 0) {
      const offeredEntityId = view.proactiveOffers[0].entityId;
      return view.context.find(e => e.id === offeredEntityId)
        ?? view.focal;
    }

    // If disambiguation is pending, that's the only thing worth showing
    if (view.focal.renderHint.disambiguationPending) {
      return view.focal;
    }

    return view.focal;
  }
}
```

## Building the three presentation branches

Each branch targets a different sensory channel. They're built in parallel but designed to work together — the haptic fires first to get attention, the display shows context, audio elaborates if the user wants more.

### Display presentation

The watch display renders one of three layouts depending on the waypoint type. The constraint is severe: no more than two lines of body text, one primary action, one optional secondary action.

```typescript
private buildDisplayPresentation(
  entity: ProjectedEntity,
  view: ProjectedView
): DisplayPresentation {

  // Disambiguation: show labelled options, nothing else
  if (entity.renderHint.disambiguationPending) {
    return {
      layout: 'option-list',
      prompt: this.shortenForWatch(
        `Which ${entity.name}?`, 28
      ),
      options: view.executionState?.candidates.map((c, i) => ({
        id: `option-${i}`,
        label: PHONETIC_LABELS[i],
        // Keep option text to one line — truncate if needed
        description: this.shortenForWatch(c.description, 32),
        action: `select_candidate_${i}`,
      })) ?? [],
    };
  }

  // Proactive offer: minimal — just enough to decide yes or no
  if (view.proactiveOffers.length > 0) {
    const offer = view.proactiveOffers[0];
    return {
      layout: 'confirmation',
      prompt: this.shortenForWatch(offer.prompt, 48),
      primaryAction:  { label: 'Yes', action: 'confirm_offer' },
      secondaryAction: { label: 'Dismiss', action: 'dismiss_offer' },
    };
  }

  // Standard entity summary
  return {
    layout: 'summary',
    // Entity name as title
    title: this.shortenForWatch(entity.name, 24),
    // One line of context — role + most relevant attribute
    body: this.summarizeEntity(entity),
    // The one thing the user can do right now
    primaryAction: {
      label: this.affordanceLabel(entity.renderHint.primaryAffordance),
      action: entity.renderHint.primaryAffordance,
    },
  };
}

private summarizeEntity(entity: ProjectedEntity): string {
  switch (entity.role) {
    case 'diagnostic':
      // "Type error · line 42"
      return `${this.humanizeSeverity(entity.attributes.severity)} · `
           + `line ${entity.attributes.line}`;

    case 'task':
      // "In progress · due today"
      return `${entity.attributes.status} · ${this.humanizeDue(entity)}`;

    case 'physical_device':
      // "Living room lamp · on, 80%"
      return `${entity.attributes.location} · ${this.describeState(entity)}`;

    case 'source_file':
      // "Modified 2 min ago"
      return `Modified ${this.timeAgo(entity.attributes.lastModified)}`;

    default:
      return entity.role.replace('_', ' ');
  }
}

private shortenForWatch(text: string, maxChars: number): string {
  if (text.length <= maxChars) return text;
  // Truncate at word boundary, add ellipsis
  return text.slice(0, maxChars - 1).replace(/\s\S+$/, '') + '…';
}
```

### Audio presentation

Watch audio is used sparingly — only when the display alone is insufficient, or when the user explicitly requests elaboration. The TTS is shorter than the voice-only path because the display is carrying visual context simultaneously.

```typescript
private buildAudioPresentation(
  entity: ProjectedEntity,
  view: ProjectedView
): AudioPresentation {

  // Proactive offer: one short spoken question
  if (view.proactiveOffers.length > 0) {
    return {
      tts: view.proactiveOffers[0].spokenPrompt,
      // Short — the display has the full text
      ssml: `<speak><s>${view.proactiveOffers[0].spokenPrompt}</s></speak>`,
      autoPlay: false,   // Only play if user raises wrist or taps
      maxDuration: 4000, // 4 seconds max — watch speaker is small
    };
  }

  // Disambiguation: read the options aloud
  if (entity.renderHint.disambiguationPending) {
    const options = view.executionState?.candidates
      .map((c, i) => `${PHONETIC_LABELS[i]}: ${c.spokenDescription}`)
      .join('. ') ?? '';

    return {
      tts: options,
      ssml: this.buildDisambiguationSSML(view.executionState!),
      autoPlay: true,    // Must play — user can't see full options on tiny display
      maxDuration: 8000,
    };
  }

  // Standard: brief spoken summary, only on explicit request
  return {
    tts: this.buildEntitySummaryUtterance(entity),
    ssml: `<speak><s>${this.buildEntitySummaryUtterance(entity)}</s></speak>`,
    autoPlay: false,
    maxDuration: 6000,
  };
}

private buildEntitySummaryUtterance(entity: ProjectedEntity): string {
  // Shorter than the voice-only path — display is doing half the work
  switch (entity.role) {
    case 'diagnostic':
      return `${this.humanizeSeverity(entity.attributes.severity)} `
           + `on line ${entity.attributes.line}.`;
    case 'task':
      return `${entity.name}. ${entity.attributes.status}.`;
    default:
      return entity.name;
  }
}
```

### Haptic presentation

Haptic is the most underused channel in ambient computing. On a watch, it's the *first* channel — it fires before the display turns on, before any audio plays. It tells the user whether to look at their wrist at all.

```typescript
private buildHapticPresentation(
  entity: ProjectedEntity,
  view: ProjectedView
): HapticPresentation {

  // Each semantic situation maps to a distinct haptic pattern
  // The user learns these patterns over time — haptic becomes a language

  if (view.proactiveOffers.length > 0) {
    return {
      // Soft, single tap — "something noticed, not urgent"
      pattern: 'notification-soft',
      timing: 'immediate',
      // If user doesn't respond in 3s, don't repeat — respect decay
      repeat: false,
    };
  }

  if (entity.renderHint.disambiguationPending) {
    return {
      // Double tap — "needs your decision"
      pattern: 'notification-double',
      timing: 'immediate',
      repeat: false,
    };
  }

  if (entity.role === 'diagnostic' && entity.attributes.severity === 'error') {
    return {
      // Firm single tap — "something broke"
      pattern: 'alert',
      timing: 'immediate',
      repeat: false,
    };
  }

  if (entity.role === 'task' && this.isDueSoon(entity)) {
    return {
      // Rising double tap — "time-sensitive"
      pattern: 'reminder',
      timing: 'immediate',
      repeat: true,
      repeatInterval: 30_000,  // Repeat every 30s until acknowledged
      repeatMax: 3,
    };
  }

  // Confirmation feedback after user action — "that worked"
  if (view.meta.trigger === 'action-completed') {
    return {
      pattern: 'success',
      timing: 'immediate',
      repeat: false,
    };
  }

  // Default: no haptic — don't tap the wrist for low-salience updates
  return { pattern: null, timing: 'none' };
}
```

The pattern library maps to named OS haptic patterns rather than raw vibration parameters — this is the only practical choice since watchOS, Wear OS, and Galaxy Watch each expose their own named pattern APIs. The `WatchRenderClient` translates `'notification-double'` to the platform-specific call.

## The expected inputs

The `expectedInputs` block defines what the watch can accept in response to this waypoint — and critically, it maps each input modality to the same semantic action, so the runtime doesn't care how the user responded:

```typescript
private buildExpectedInputs(
  entity: ProjectedEntity,
  caps: WatchCapabilities
): ExpectedInputs {

  const inputs: ExpectedInputs = { voice: null, touch: null, gesture: null };

  // Voice input — if mic available
  if (caps.audio.input) {
    inputs.voice = {
      // Constrained grammar based on context
      grammar: entity.renderHint.disambiguationPending
        ? {
            type: 'constrained',
            phrases: this.buildDisambiguationPhrases(
              view.executionState!.candidates
            ),
          }
        : { type: 'open' },
      triggerPhrase: null,   // Tap to talk on watch, no wake word
    };
  }

  // Touch — tap targets on display
  inputs.touch = {
    events: this.mapActionsToTouchTargets(entity),
  };

  // Crown / swipe — for scrolling option lists
  if (entity.renderHint.disambiguationPending) {
    inputs.gesture = {
      crown: { direction: 'scroll', action: 'scroll_options' },
      swipe: { direction: 'up', action: 'dismiss_offer' },
    };
  }

  return inputs;
}
```

## The render client on the watch

Like the XR client, the watch render client is intentionally dumb. It receives the `InteractionWaypoint` and calls platform APIs:

```typescript
// apps/watch-client/src/render-client.ts

export class WatchRenderClient {

  async render(waypoint: InteractionWaypoint) {

    const { presentations, expectedInputs } = waypoint;

    // 1. Haptic fires first — before display wakes
    if (presentations.haptic?.pattern) {
      await this.hapticEngine.play(
        this.mapPattern(presentations.haptic.pattern)
      );
    }

    // 2. Display update
    if (presentations.display) {
      this.displayRenderer.render(presentations.display);
    }

    // 3. Audio — only if autoPlay, otherwise wait for user action
    if (presentations.audio?.autoPlay) {
      await this.ttsEngine.speak(
        presentations.audio.ssml,
        presentations.audio.maxDuration
      );
    }

    // 4. Arm input handlers
    this.inputHandler.arm(expectedInputs, waypoint.id);
  }

  private mapPattern(pattern: HapticPatternName): PlatformHaptic {
    // Translates semantic pattern names to platform-specific APIs
    const map: Record<HapticPatternName, PlatformHaptic> = {
      'notification-soft':   WKHapticType.notification,
      'notification-double': WKHapticType.directionUp,
      'alert':               WKHapticType.failure,
      'reminder':            WKHapticType.retry,
      'success':             WKHapticType.success,
    };
    return map[pattern];
  }
}
```

## A concrete end-to-end example

The user is away from their desk. A Worker Agent has finished phase one of the database migration and is requesting approval. Here's the full path to the wrist:

The `TopologicalExecutor` marks the agent's batch of `EditIntents` as `pending_approval`. The `ProjectionEngine` builds a `ProjectedView` with `executionState.type: 'approval-pending'` and `proactiveOffers` containing a summary of what the agent wants to do.

The `WatchProjector` receives this view and builds a waypoint. The haptic branch emits `notification-double` — double tap, needs a decision. The display branch renders a `confirmation` layout: *"Migration phase 1 ready — 4 files modified."* with Yes and Dismiss actions. The audio branch prepares *"The migration agent wants to update four files. Approve?"* but sets `autoPlay: false` — the display is enough unless the user taps for more.

The watch fires the double tap. The user glances down. They tap Yes. The `WatchRenderClient` sends `{ waypointId, action: 'confirm_offer' }` to the phone. The phone's runtime maps this to a `ConfirmIntent`, which the `TopologicalExecutor` uses to approve and execute the pending `EditIntents`. The watch fires the `success` haptic — single clean tap. The transaction is done.

No notification tray. No app to open. No context switch back to the desk. The entire approval happened in four seconds on a 44mm display.

## The design principle underneath all of this

The watch forces a clarity that every other surface can hide from. You cannot put a complex UI on a watch. You cannot play a two-minute audio explanation. You cannot render a spatial overlay. You are forced to answer the question every USIR waypoint should answer but often avoids:

*What is the single most important thing this person needs to know or do right now?*

The watch doesn't render USIR interactions well because it's powerful. It renders them well because it refuses to render anything else.
