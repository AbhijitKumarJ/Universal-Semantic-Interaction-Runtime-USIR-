 Good question — this is where the architectural philosophy meets actual pixels. The key principle is that the runtime **never renders**. It projects state; the presentation layer renders it.

---

## The projection pipeline

There are three distinct stages: **snapshot → projection → render hint → client render**.

The runtime is only responsible for the first two. The screen client owns the last two.

```
SemanticSnapshot
      │
      ▼
  ProjectionEngine          ← lives in @usir/runtime
  (filters, ranks, scopes)
      │
      ▼
  ProjectedView             ← a typed, device-aware subset of the snapshot
      │
      ▼
  RenderHint                ← layout suggestions, not mandates
      │
      ▼
  Screen Client             ← lives in @usir/vscode-ext, @usir/browser, etc.
  (owns all DOM/pixel work)
```

---

## The `ProjectedView` type

The snapshot contains *everything* — the entire semantic graph. A screen client only needs a slice of it, filtered to what's relevant to the current focus and device capability.

```typescript
// packages/protocol/src/projection/index.ts

export interface ProjectedView {
  // The primary entity in focus — always present
  focal: ProjectedEntity;
  
  // Contextually relevant neighbours (relations, siblings, dependents)
  context: ProjectedEntity[];
  
  // Active execution state — pending plans, agent tasks, ambiguities
  executionState: ExecutionState | null;
  
  // Pending proactive offers waiting for user acknowledgement
  proactiveOffers: ProactiveOffer[];
  
  // Metadata about the projection itself
  meta: ProjectionMeta;
}

export interface ProjectedEntity {
  id: string;
  name: string;
  role: EntityRole;
  
  // Rendered representation — the runtime suggests, client decides
  renderHint: RenderHint;
  
  // Available intents on this entity, pre-filtered by authorization
  affordances: AffordanceSet;
  
  // Provenance summary — who last touched this and why
  provenance?: ProvenanceSummary;
}

export interface RenderHint {
  // Suggested layout role for this entity
  layout: 'primary' | 'sidebar' | 'inline' | 'ambient' | 'hidden';
  
  // Visual weight — relative importance in current context
  salience: number;         // 0.0 – 1.0
  
  // Suggested interaction affordance to make visible
  primaryAffordance: string; // e.g. 'edit', 'navigate', 'execute', 'explain'
  
  // Whether the disambiguation UI should be shown for this entity
  disambiguationPending: boolean;
}
```

## How the `ProjectionEngine` builds the view

The engine runs on every snapshot update. It has three jobs: **scope**, **rank**, and **annotate**.

```typescript
// packages/runtime/src/projection/engine.ts

export class ProjectionEngine {

  project(snapshot: SemanticSnapshot): ProjectedView {
    
    // 1. SCOPE — which entities are relevant right now?
    const candidates = this.scope(snapshot);
    
    // 2. RANK — of those, which matter most?
    const ranked = this.rank(candidates, snapshot.hotTier);
    
    // 3. ANNOTATE — attach render hints and affordances
    const projected = ranked.map(e => this.annotate(e, snapshot));
    
    return {
      focal: projected[0],
      context: projected.slice(1),
      executionState: this.extractExecutionState(snapshot),
      proactiveOffers: this.extractProactiveOffers(snapshot),
      meta: {
        deviceClass: snapshot.meta.deviceClass,
        projectedAt: Date.now(),
        snapshotVersion: snapshot.version,
      }
    };
  }

  private scope(snapshot: SemanticSnapshot): SemanticEntity[] {
    const hot = snapshot.hotTier;
    
    // Always include the active entity
    const focal = snapshot.entityGraph.get(hot.activeEntityId);
    
    // Include immediate relations (e.g. file → its functions, error → its file)
    const neighbours = this.walkRelations(focal, snapshot.entityGraph, depth: 1);
    
    // Include anything the user has touched in the last 30 seconds
    const recent = snapshot.interactionMemory.getRecent(30_000)
      .map(id => snapshot.entityGraph.get(id))
      .filter(Boolean);
    
    return deduplicate([focal, ...neighbours, ...recent]);
  }

  private rank(
    entities: SemanticEntity[],
    hot: HotTier
  ): SemanticEntity[] {
    return entities.sort((a, b) => {
      // Focal entity always first
      if (a.id === hot.activeEntityId) return -1;
      if (b.id === hot.activeEntityId) return 1;
      
      // Entities with active errors rank higher
      const aErr = a.role === 'diagnostic' ? 1 : 0;
      const bErr = b.role === 'diagnostic' ? 1 : 0;
      if (aErr !== bErr) return bErr - aErr;
      
      // Recency as tiebreaker
      return b.lastTouched - a.lastTouched;
    });
  }

  private annotate(
    entity: SemanticEntity,
    snapshot: SemanticSnapshot
  ): ProjectedEntity {
    
    const isFocal = entity.id === snapshot.hotTier.activeEntityId;
    const hasError = entity.role === 'diagnostic';
    const isPending = snapshot.executionPlan?.pendingEntityIds.includes(entity.id);

    return {
      ...entity,
      renderHint: {
        layout: isFocal ? 'primary' : hasError ? 'sidebar' : 'inline',
        salience: isFocal ? 1.0 : hasError ? 0.8 : 0.4,
        primaryAffordance: hasError ? 'explain' : isFocal ? 'edit' : 'navigate',
        disambiguationPending: isPending ?? false,
      },
      affordances: this.computeAffordances(entity, snapshot.authorization),
      provenance: this.summarizeProvenance(entity.id),
    };
  }
}
```

## What the screen client does with it

The screen client subscribes to `ProjectedView` updates and maps them to DOM. Critically, the client is **allowed to ignore render hints** — they are suggestions, not commands. A VS Code extension respects `layout: 'sidebar'` differently than a browser extension would.

```typescript
// apps/vscode-extension/src/renderer.ts

export class ScreenProjectionRenderer {

  onProjectedViewUpdate(view: ProjectedView) {
    
    // 1. Render the focal entity
    this.renderFocal(view.focal);
    
    // 2. Render context panel (sidebar, inline decorators)
    this.renderContext(view.context);
    
    // 3. Handle any pending disambiguation
    if (view.focal.renderHint.disambiguationPending) {
      this.renderDisambiguationUI(view.executionState);
    }
    
    // 4. Show proactive offers if any
    view.proactiveOffers.forEach(offer => this.renderProactiveNudge(offer));
  }

  private renderFocal(entity: ProjectedEntity) {
    switch (entity.renderHint.primaryAffordance) {
      case 'edit':
        // Highlight the entity in the editor, show inline action buttons
        this.decorateActiveEntity(entity);
        break;
        
      case 'explain':
        // Paint the diagnostic highlight, show "Trace this?" nudge
        this.decorateDiagnostic(entity);
        break;
        
      case 'navigate':
        // Show a breadcrumb or jump target
        this.decorateNavTarget(entity);
        break;
    }
  }

  private renderDisambiguationUI(state: ExecutionState) {
    // Paint phonetic labels ("Alpha", "Bravo") over each candidate entity
    // The user's next voice command or keypress resolves the ambiguity
    state.candidates.forEach((candidate, i) => {
      this.paintLabel(candidate.entityId, PHONETIC_LABELS[i]);
    });
  }
}
```

## The salience gradient in practice

The `salience` field is what makes the screen feel *responsive to context* rather than just rendering everything at equal weight. A few concrete examples of how it flows through:

When the user is **in flow** — typing, no errors, clear focus — the projection engine produces one high-salience focal entity and everything else fades to `salience < 0.3`. The screen client dims non-focal decorators, collapses the context panel. The interface gets out of the way.

When the user **hits an error** — typing halts, cursor dwells on a diagnostic — the error entity's salience jumps to `0.8` even though it's not the focal entity. The screen client notices this and promotes it to `layout: 'sidebar'` without the user asking. This is the proactive nudge mechanism expressed purely through projection, no special-casing needed.

When **disambiguation is pending** — the execution plan has an `UNRESOLVED` sentinel — the screen client detects `disambiguationPending: true` on multiple entities simultaneously, and knows to render the Visual Handshake overlay. The runtime doesn't tell the client *how* to render disambiguation; it just signals that it's needed, and the client implements it natively for its platform.

## Why this separation matters

The cleanest thing about this design is what it prevents. The runtime has zero knowledge of:

- CSS, DOM, WebGL, or any rendering API
- Whether the client is VS Code, a browser, an XR overlay, or a terminal
- What font size the user prefers or whether they have a 4K monitor

And the screen client has zero knowledge of:

- How the semantic graph was built
- Which LLM classified the intent
- Whether the active entity came from a native adapter or a VLM-compiled Zero-Shot adapter

The `ProjectedView` is the contract. Everything on the left side of it is the runtime's problem. Everything on the right side is the client's problem. Changing the VS Code renderer doesn't touch the projection engine, and improving the ranking algorithm doesn't require any client updates.

That clean boundary is what makes it possible to add a new presentation surface — say, a terminal TUI client, or an XR overlay — without touching any existing code. The new client just subscribes to the same `ProjectedView` stream and renders it however makes sense for its medium.
