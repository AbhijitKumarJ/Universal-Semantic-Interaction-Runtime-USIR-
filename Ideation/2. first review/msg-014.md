# Message 14 — USIR

_Sent at: 2026-06-01 01:13:03_


 Federation is where the entire USIR architecture either holds together or falls apart. Every previous layer assumed one runtime, one user, one semantic graph. Federation means two sovereign runtimes — each with its own entity graph, its own provenance log, its own trust model — need to collaborate on shared entities without either instance subordinating itself to the other.

The design principle that makes this tractable is the same one that runs through the whole system: **share semantics, not state**.

---

## What "federated" actually means here

Two USIR instances federating is not synchronisation in the traditional sense. It is not:

- A shared database with two clients
- One instance replicating its graph to the other
- A central relay server that both instances talk through

It is two independent runtimes that agree to treat a **subset of their entities as jointly owned**, where mutations from either side propagate to the other as `IntentEnvelopes` — the same primitives that local user actions use. The receiving runtime applies those intents through its own `TopologicalExecutor`, under its own trust rules. Neither runtime has privileged access to the other's full graph.

## The connection lifecycle

```typescript
// packages/runtime/src/federation/connection-manager.ts

export class FederationConnectionManager {

  async initiate(
    shareIntent: ShareIntent,
    localSnapshot: SemanticSnapshot
  ): Promise<FederatedSession> {

    // 1. Resolve the peer's identity
    const peerId = await this.identityResolver.resolve(shareIntent.collaboratorId);
    // collaboratorId is a DID — decentralised identity, no central auth

    // 2. Establish WebRTC data channel
    const channel = await this.webrtc.connect(peerId, {
      ordered: true,        // Intent ordering matters
      maxRetransmits: 3,    // Reliable delivery, not lossy
    });

    // 3. Perform capability handshake — what can each side do?
    const peerCaps = await this.handshake(channel, localSnapshot);

    // 4. Negotiate the shared sub-graph
    const sharedGraph = await this.negotiate(
      shareIntent,
      localSnapshot,
      peerCaps
    );

    // 5. Bootstrap the peer's CRDT with our current state
    await this.bootstrap(channel, sharedGraph, localSnapshot);

    return new FederatedSession(channel, sharedGraph, peerCaps);
  }

  private async handshake(
    channel: RTCDataChannel,
    snapshot: SemanticSnapshot
  ): Promise<PeerCapabilities> {

    const localHello: HelloMessage = {
      type: 'hello',
      protocolVersion: USIR_FEDERATION_VERSION,
      identity: this.localIdentity.did,
      // Declare what this instance can do
      capabilities: {
        modalities: snapshot.meta.activeModalities,
        intentLayers: ['L1', 'L2', 'L3', 'L4', 'L5', 'L6', 'L7', 'L8'],
        crdtImplementation: 'yjs-v1',
        maxSharedEntities: 500,
      },
      // Proof of identity — signed with local private key
      signature: await this.localIdentity.sign(snapshot.meta.instanceId),
    };

    channel.send(JSON.stringify(localHello));

    const peerHello = await this.awaitMessage<HelloMessage>(channel, 'hello');

    // Verify the peer's signature before proceeding
    await this.verifyPeerIdentity(peerHello);

    return peerHello.capabilities;
  }

  private async negotiate(
    intent: ShareIntent,
    snapshot: SemanticSnapshot,
    peerCaps: PeerCapabilities
  ): Promise<SharedSubGraph> {

    // The ShareIntent specifies what to share and with what permissions
    const requestedEntities = Array.isArray(intent.target)
      ? intent.target
      : [intent.target];

    // Expand to include required relations
    // (can't share a function without sharing its file)
    const expanded = this.expandToClosedSubGraph(
      requestedEntities,
      snapshot.entityGraph
    );

    // Apply permission model
    const permissionedGraph = this.applyPermissions(
      expanded,
      intent.permissions
    );

    return {
      entityIds: permissionedGraph.map(e => e.id),
      permissions: intent.permissions,
      // Each entity gets a CRDT document
      crdtDocs: this.initialiseCRDTs(permissionedGraph),
    };
  }
}
```

## The CRDT layer

CRDTs (Conflict-free Replicated Data Types) are the foundation that makes concurrent edits safe. But USIR doesn't use CRDTs the way a text editor does — it doesn't CRDT the raw content of files. It CRDTs the **semantic entity graph**: the nodes, their attributes, and their relations.

```typescript
// packages/runtime/src/federation/crdt-graph.ts
import * as Y from 'yjs';

export class FederatedEntityGraph {

  private doc: Y.Doc;
  private entities: Y.Map<SerializedEntity>;
  private relations: Y.Map<Y.Array<string>>;

  constructor(sharedGraph: SharedSubGraph) {
    this.doc = new Y.Doc();
    this.entities = this.doc.getMap('entities');
    this.relations = this.doc.getMap('relations');

    // Initialise with our current state
    this.doc.transact(() => {
      for (const entity of sharedGraph.crdtDocs) {
        this.entities.set(entity.id, this.serialize(entity));
      }
    });
  }

  // Apply a local intent — came from our own user
  applyLocalIntent(intent: IntentEnvelope): Y.Update {
    let update: Y.Update;

    this.doc.transact(() => {
      update = this.mutateFromIntent(intent);
    });

    // Return the binary CRDT update to broadcast to peers
    return Y.encodeStateAsUpdate(this.doc);
  }

  // Apply a remote update — came from a peer's intent
  applyRemoteUpdate(update: Y.Update): void {
    // Yjs handles all conflict resolution automatically
    // Last-write-wins on leaf values, union on sets
    Y.applyUpdate(this.doc, update);
  }

  // The critical constraint: CRDTs track what changed,
  // but USIR tracks WHY it changed via provenance
  private mutateFromIntent(intent: IntentEnvelope): Y.Update {
    switch (intent.intent.type) {

      case 'intent.manipulation.edit': {
        const entity = this.entities.get(intent.intent.target as string);
        if (!entity) throw new EntityNotFoundError(intent.intent.target);

        // Store the intent ID alongside the mutation
        // so the provenance log can reconstruct causality
        this.entities.set(intent.intent.target as string, {
          ...entity,
          attributes: this.applyEdit(entity.attributes, intent.intent),
          lastMutatedBy: intent.actor,
          lastMutationIntentId: intent.intentId,
        });
        break;
      }

      case 'intent.manipulation.create': {
        const newEntity = this.createFromIntent(intent.intent);
        this.entities.set(newEntity.id, this.serialize(newEntity));
        break;
      }

      case 'intent.manipulation.delete': {
        // Never hard-delete in a federated graph
        // Tombstone instead — peer needs to know this entity is gone
        const entity = this.entities.get(intent.intent.target as string);
        this.entities.set(intent.intent.target as string, {
          ...entity,
          tombstoned: true,
          tombstonedBy: intent.actor,
          tombstonedAt: Date.now(),
        });
        break;
      }
    }

    return Y.encodeStateAsUpdate(this.doc);
  }
}
```

The tombstone pattern for deletions is important. If User A deletes an entity at the same moment User B edits it, a hard delete would silently discard User B's edit. A tombstone preserves the causal history and lets the two runtimes surface a conflict resolution waypoint rather than silently losing data.

## The `FederatedSession` — the live connection

Once the connection is established and the CRDT graph is bootstrapped, the `FederatedSession` manages the live exchange:

```typescript
// packages/runtime/src/federation/session.ts

export class FederatedSession {

  constructor(
    private channel: RTCDataChannel,
    private sharedGraph: SharedSubGraph,
    private peerCaps: PeerCapabilities,
    private crdtGraph: FederatedEntityGraph
  ) {
    this.channel.onmessage = e => this.onPeerMessage(JSON.parse(e.data));
  }

  // Called by our TopologicalExecutor when a local intent
  // touches a shared entity
  async onLocalIntentExecuted(intent: IntentEnvelope): Promise<void> {

    // Only broadcast if this intent touched a shared entity
    if (!this.touchesSharedEntity(intent)) return;

    // 1. Apply to our local CRDT, get the update delta
    const update = this.crdtGraph.applyLocalIntent(intent);

    // 2. Build the federation message
    const message: FederationMessage = {
      type: 'intent-update',
      intentEnvelope: intent,
      crdtUpdate: Array.from(update),   // Binary, serialised as array
      senderState: {
        instanceId: this.localIdentity.instanceId,
        vectorClock: this.vectorClock.increment(),
      },
      signature: await this.localIdentity.sign(intent.intentId),
    };

    this.channel.send(JSON.stringify(message));
  }

  private async onPeerMessage(message: FederationMessage): Promise<void> {
    switch (message.type) {

      case 'intent-update':
        await this.handlePeerIntent(message);
        break;

      case 'cursor-position':
        await this.handlePeerCursor(message);
        break;

      case 'presence-update':
        await this.handlePeerPresence(message);
        break;

      case 'disconnect':
        await this.handlePeerDisconnect(message);
        break;
    }
  }

  private async handlePeerIntent(message: FederationMessage): Promise<void> {

    // 1. Verify the message signature
    await this.verifySignature(message);

    // 2. Check causal ordering — do we have all predecessors?
    if (!this.vectorClock.canApply(message.senderState.vectorClock)) {
      // We're missing earlier updates — request them before applying
      await this.requestMissingUpdates(message.senderState);
      await this.pendingQueue.enqueue(message);
      return;
    }

    // 3. Apply the CRDT update — handles concurrent edits automatically
    this.crdtGraph.applyRemoteUpdate(
      new Uint8Array(message.crdtUpdate)
    );

    // 4. Feed through our local TopologicalExecutor
    // This is the key insight: peer intents are executed locally,
    // not blindly applied. Our own trust rules govern the execution.
    await this.topologicalExecutor.executeFederated(
      message.intentEnvelope,
      {
        actor: 'peer',
        peerId: message.senderState.instanceId,
        permissions: this.sharedGraph.permissions,
      }
    );

    // 5. Record to provenance with full causal lineage
    await this.provenanceStore.record({
      intentId: message.intentEnvelope.intentId,
      actor: {
        type: 'peer',
        id: message.senderState.instanceId,
      },
      rationale: {
        type: 'delegated',
        interpretedIntent: message.intentEnvelope.intent.instruction,
      },
      authorization: {
        type: 'direct',
        allowedEntityIds: this.sharedGraph.entityIds,
        signature: message.signature,
      },
      causalParents: [message.senderState.vectorClock.lastIntentId],
      timestamp: Date.now(),
      contentHash: await this.hashSharedGraph(),
    });

    // 6. Update the vector clock
    this.vectorClock.update(message.senderState.vectorClock);
  }
}
```

Step 4 is the architectural keystone. Peer intents don't bypass the local `TopologicalExecutor` — they flow through it, subject to the local runtime's own trust rules. If User A has `permissions: 'view'` on the shared entity, the `TopologicalExecutor` will refuse a `EditIntent` from User A's instance even if it arrives over the federation channel. The permission model is enforced locally, not trusted from the peer.

## Asymmetric modality — the scenario that proves the design

The car/desk pair-programming scenario from Part 5 of the blog series is now implementable. User B at a desk highlights a block of code and says *"what do you think of this auth logic?"*

On User B's side:

```typescript
        + "lines 42 to 58, and asking for your thoughts. "

  }],
  expectedResponses: { type: 'open' },

```

User A responds: *"That's a bug. Add a check for the audience claim."*

User A's STT captures this. A's IntentRouter maps it to an `EditIntent` targeting `file://auth.ts`. A's `TopologicalExecutor` executes it locally and broadcasts it via `FederatedSession`. User B's runtime receives the `intent-update`, applies the CRDT update, and their screen shows the code rewriting itself — with a spatial XR indicator showing User A's "presence" at that line.

Neither runtime ever saw raw audio from the other side. Neither runtime knew the other's modality. The `DiscussIntent` and `EditIntent` are the same semantic primitives regardless of whether they originated from a keyboard, a voice command, or a gaze-and-pinch gesture. The federation layer is modality-blind by design.

## Handling disconnection gracefully

WebRTC connections drop. The protocol needs to handle this without corrupting the shared graph:

```typescript
private async handlePeerDisconnect(message: FederationMessage): Promise<void> {

  // 1. Mark shared entities as 'offline-peer-owned'
  // They remain visible but show a staleness indicator
  for (const entityId of this.sharedGraph.entityIds) {
    const entity = this.crdtGraph.getEntity(entityId);
    if (entity?.lastMutatedBy === message.senderState.instanceId) {
      await this.localSnapshot.markEntityStale(entityId, {
        reason: 'peer-disconnected',
        lastSyncedAt: this.vectorClock.lastSyncTimestamp,
      });
    }
  }

  // 2. Queue any local intents that touched shared entities
  // They'll be held and replayed when the peer reconnects
  this.offlineQueue.activate(this.sharedGraph.entityIds);

  // 3. Surface a presence indicator to the user
  await this.surfaceRouter.push(
    this.buildPresenceWaypoint('peer-disconnected', message.senderState)
  );
}

async onPeerReconnect(channel: RTCDataChannel): Promise<void> {

  // Exchange state vectors to find what each side missed
  const localVector  = this.vectorClock.current();
  const peerVector   = await this.exchangeVectors(channel);

  // Compute the diff — what updates does each side need?
  const missingLocal = this.crdtGraph.getUpdatesSince(peerVector);
  const missingPeer  = await this.requestUpdatesSince(channel, localVector);

  // Apply what we missed
  for (const update of missingPeer) {
    this.crdtGraph.applyRemoteUpdate(new Uint8Array(update.crdtUpdate));
  }

  // Send what the peer missed
  channel.send(JSON.stringify({
    type: 'catch-up',
    updates: missingLocal.map(u => Array.from(u)),
  }));

  // Drain the offline queue
  for (const intent of this.offlineQueue.drain()) {
    await this.onLocalIntentExecuted(intent);
  }
}
```

The vector clock is what makes catch-up safe. Each side knows exactly which intents it has processed, so on reconnection the two runtimes can compute precisely what they each missed and exchange only those updates — no full graph resync needed.

## The trust boundary that completes the picture

The federation protocol has one rule that sits above all the others: **no runtime ever executes a peer intent without verifying its signature and checking it against the local permission model**.

This means:

A peer with `permissions: 'view'` can send `EditIntent` envelopes — they will be cryptographically valid and well-formed — but the receiving runtime's `TopologicalExecutor` will refuse to execute them. The permission check happens locally, not at the sender.

A peer with `permissions: 'edit'` can send `DeleteIntent` envelopes — but because delete is a Tier 3 irreversible action in the trust classifier, it gets routed through the checkpoint path regardless of the permission level. The user gets a disambiguation waypoint: *"User B wants to delete auth.ts. Approve?"*

A compromised peer — one whose WebRTC channel was intercepted — cannot inject arbitrary mutations, because every `FederationMessage` is signed with the peer's private key. A message that fails signature verification is dropped before it reaches the CRDT layer.

The result is a collaboration model where two sovereign runtimes cooperate without either one trusting the other unconditionally. Not because they're adversarial — but because the architecture that requires explicit trust boundaries is the same architecture that survives when trust is violated.
