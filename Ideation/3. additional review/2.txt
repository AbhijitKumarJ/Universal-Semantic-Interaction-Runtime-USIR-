**USIR**  
**Universal Semantic Interaction Runtime**  
**Towards a Post-GUI Computing Architecture**

---

### Executive Summary

Modern computing is trapped in the GUI paradigm: applications are built around pixels, buttons, menus, and screens optimized for mice and keyboards. This creates massive redundancy, poor accessibility, device fragmentation, and data silos.

**USIR** is a semantic operating layer that sits between humans (and agents) and software. It decouples **human intent** from **visual presentation** and **application implementation**. Instead of forcing users into app-specific UIs, USIR operates on a shared **semantic graph** using a **Universal Intent Ontology**, rich **Interaction Memory**, and pluggable **Semantic Adapters**.

The result is:
- Truly ambient, modality-agnostic interaction (voice, screen, XR, haptic, gesture, etc.)
- User-sovereign Personal Semantic Cloud
- Safe, auditable agentic delegation
- Federated collaboration across sovereign runtimes
- A foundation for a post-application capability marketplace

USIR does not replace existing software immediately. It progressively **semanticizes** it through adapters while providing immediate value as a powerful IDE copilot.

**Core Thesis**:  
Software should expose *meaning*, not presentation. The runtime should understand *intent*, *context*, *memory*, and *relationships*.

**Exploring CRDTs for Semantic Graphs in USIR**

CRDTs (Conflict-free Replicated Data Types) are a natural fit for USIRâ€™s **federated semantic graphs**, enabling P2P collaboration between sovereign runtimes without a central server. They guarantee **eventual consistency** even with concurrent edits, network partitions, or offline work â€” exactly whatâ€™s needed for asymmetric modality collaboration, agent delegation, and multi-user sub-graph sharing.

### Why CRDTs Matter for USIR Semantic Graphs

USIRâ€™s semantic graph consists of:
- **Nodes** (`SemanticEntity`): Files, functions, tasks, physical devices, agents, etc.
- **Edges/Relations**: `dependsOn`, `locatedIn`, `createdBy`, `references`, etc.
- **Attributes**: Provenance, metadata, render hints, capabilities.
- **Dynamic aspects**: Hot/Warm/Cold tiers, Interaction Memory references, execution plans.

In federation, multiple USIR instances share **sub-graphs** and apply intents concurrently. Traditional locking or central coordination would break the ambient, sovereign, offline-first vision. CRDTs solve this by making operations **commutative** (order-independent) or by providing deterministic merge rules.

### Challenges of Applying CRDTs to Graphs

Graphs are harder than text/lists/maps because:
- **Tombstones & Deletes**: Deleting a node/edge must not silently break concurrent additions (use tombstones or version vectors).
- **Structural Integrity**: Maintaining DAG invariants or preventing cycles during merges.
- **Reference Stability**: "That file" or spatial references must remain valid after concurrent changes (Interaction Memory helps here).
- **Metadata Bloat**: Each node/edge may need unique IDs, causal metadata, or provenance.
- **Partial Replication**: USIR often shares only sub-graphs, not the entire world graph.

Recent research (e.g., Directed Acyclic Graph CRDTs) and libraries address many of these.

### Recommended CRDT Approaches for USIR

#### 1. **Hybrid / Composable Strategy (Recommended)**
Donâ€™t CRDT the entire raw graph. Instead:

- Use **Yjs** (or Loro/Automerge) for the **structural backbone** (Maps for entities, arrays/trees for relations).
- Overlay **semantic intent application** on top: Intents are applied through the deterministic `TopologicalExecutor`, which then mutates the CRDT structures.
- Store **Provenance** and heavy metadata in a separate append-only log (not fully CRDTâ€™d, but versioned and synced).

**Why this wins**:
- Leverages mature libraries for low-level sync.
- Keeps high-level semantics (intents, provenance) under USIR control for safety/auditability.
- Matches the existing design where peer intents flow through the local executor.

#### 2. Popular Libraries Evaluated

| Library     | Strengths for USIR                              | Weaknesses                          | Best For |
|-------------|------------------------------------------------|-------------------------------------|----------|
| **Yjs**     | Mature, fast, excellent ecosystem (WebRTC, providers), SharedMap + custom types, strong community | Less "JSON-native" than Automerge   | General federation, real-time |
| **Automerge** | True JSON-like documents, great for offline-first, good memory improvements in v2/v3 | Slightly heavier for very large graphs | Personal Cloud state |
| **Loro**    | High performance (Eg-walker), supports trees/movable lists, modern | Younger ecosystem                   | Performance-critical spatial/XR |
| **Custom Graph CRDT** | Full control over semantic rules (e.g., DAG CRDTs) | High implementation effort          | Long-term core |

**Recommendation**: Start with **Yjs** for the MVP federation layer (as sketched in previous blogs). Itâ€™s battle-tested for collaborative apps and has good support for custom shared types.

#### 3. Concrete Implementation Patterns for Semantic Graphs

**Entity Storage**:
```typescript
// Using Yjs
const entities = doc.getMap<SerializedEntity>('entities');
const relations = doc.getMap<Y.Array<string>>('relations'); // or adjacency list
```

**Applying an Intent** (local or from peer):
```typescript
function applyEditIntent(intent: EditIntent, graph: FederatedEntityGraph) {
  doc.transact(() => {
    const entity = entities.get(intent.target);
    if (!entity) return;

    // Semantic mutation (business logic)
    const updated = executeEdit(entity, intent.change);

    entities.set(intent.target, updated);

    // Record provenance (separate structure or embedded)
    provenanceLog.push(createProvenance(intent));
  });
}
```

**Tombstones for Safe Deletes**:
Never hard-delete in shared sub-graphs. Mark as `tombstoned: true` + timestamp + actor. Peers can clean up tombstones after consensus (via vector clocks or explicit GC intents).

**Interaction Memory Integration**:
- Store recent references as a CRDT List or Map with stable IDs.
- Use spatial indexing (e.g., R-trees or simple bounding volumes in XR) that can be partially synced.

**Sub-Graph Sharing**:
- On `ShareIntent`, create a new Yjs document (or sub-document) containing only the projected sub-graph.
- Use Yjs sub-documents or scoping to isolate shared parts while keeping private state local.

### Benefits for USIR

- **True P2P Federation**: Works offline; merges cleanly on reconnect (vector clocks handle ordering).
- **Asymmetric Collaboration**: Changes from voice user or XR user converge naturally.
- **Agent Safety**: Sandboxed sub-graphs can be CRDT-backed with strict permissions enforced locally.
- **Provenance Synergy**: CRDT updates can embed or reference provenance records for full audit trails.
- **Scalability**: Only sync deltas; Hot Tier remains fast and local.

### Trade-offs & Mitigations

- **Metadata Overhead**: Mitigate with aggressive pruning of old tombstones + compression.
- **Merge Semantics**: Define clear "last meaningful win" or domain-specific rules (e.g., via intents) so merges feel intuitive.
- **Performance on Large Graphs**: Use partial replication + tiered syncing (only sync Hot/Warm tiers in real-time).
- **Complexity**: Start simple (Yjs Maps + Lists) and evolve toward specialized graph CRDTs.

### Open Research Areas

- Dedicated **Semantic Graph CRDT** that understands ontologies and provenance natively.
- Efficient **partial replication** of sub-graphs with causal consistency.
- Combining CRDTs with **vector embeddings** for semantic search within the graph.
- Garbage collection strategies that respect provenance history.

### Capability Levels
- **Full** â€” Adapter can execute the intent reliably and return proper results.

- **None** â€” Not supported (router should reject or suggest alternatives).

### **Detailed Adapter Profiles**

#### **1. VS Code / IDE Native Adapter** (Strongest starting point)
**Best at**: Code-centric workflows  
**Strength**: Full support for L4â€“L7 (manipulation, creation, delegation)  
**Weakness**: Limited physical world awareness

- **Full Support**: Almost everything code-related (Edit, Refactor, Execute, Test, Scaffold, Delegate)
- **Partial**: Physical/IoT intents
- **Special Power**: Direct LSP + Extension API access â†’ extremely low latency and high precision

#### **2. Browser / DOM Adapter**
**Best at**: Web applications, dashboards, design tools  
**Strength**: Excellent L1â€“L3 + Collaboration

- **Full**: Navigation, Focus, Information, Collaboration
- **Partial**: Edit (via DOM mutation), Create (form filling)
- **None**: Deep code execution, OS-level actions
- **Technique**: Uses Accessibility Tree + DOM queries + Playwright-style automation

#### **3. OS Accessibility Adapter** (Windows UIA, macOS AX, Linux AT-SPI)
**Best at**: Legacy desktop apps  
**Strength**: Broad but shallow coverage

- **Full**: Basic navigation and focus
- **Partial**: Simple clicks, text reading, basic forms
- **None**: Complex refactoring or code execution
- **Limitation**: Very brittle; relies heavily on Zero-Shot VLM fallback

#### **4. Zero-Shot VLM Compiler Adapter**
**Role**: Universal fallback / legacy app bridge  
**Usage**: JIT compilation of unknown UIs into semantic hooks

- **Full**: Initial semantic mapping
- **Partial**: Most manipulation after compilation
- **None**: Real-time continuous actions (too slow)
- **Key Rule**: Used for discovery + compilation, then hands off to deterministic hooks

#### **5. XR / Spatial Adapter**
**Best at**: 3D, ambient, physical-digital fusion

- **Full**: Spatial navigation, focus (gaze), 3D manipulation, collaboration
- **Strong**: L1, L2, L3, L8
- **Partial**: Creation & editing of digital twins
- **Unique**: Raycasting + gesture support for "that one over there"

#### **6. IoT / Physical Device Adapter**
**Best at**: Real-world control

- **Full**: Simple commands on known devices (lights, thermostats)
- **Partial**: Complex queries involving physical context
- **None**: Code manipulation, deep analysis

### **Capability Matching Rules (for Intent Router)**

```typescript
interface AdapterCapability {
  adapter: string;
  level: 'Full' | 'Partial' | 'None';
  confidence: number;        // 0.0 - 1.0
  requirements?: string[];   // e.g. ["lsp_ready", "a11y_tree_available"]
}

function getBestAdapter(intent: ClassifiedIntent, snapshot: SemanticSnapshot): AdapterCapability {
  const candidates = availableAdapters
    .map(adapter => evaluateCapability(adapter, intent, snapshot))
    .filter(c => c.level !== 'None')
    .sort((a, b) => b.confidence - a.confidence);

  return candidates[0];
}
```
